## [0.1.0] - 2026-06-26

First public release.

### Added
- `introspect` skill: scans a target repo's stack and generates a tailored harness
  (root `CLAUDE.md` spine ‚Äî including an explicit `## Agents` routing block so the
  main agent delegates to the right agent without being named ‚Äî + stack `*-architect`
  agent + `.claude/harness-kit.json` verify config + specs/ADR scaffolding).
- `detect.sh` detection engine with a 23-case test suite ‚Äî layered-precedence
  detection of language / framework / test-runner / package-manager / monorepo /
  data-layer, plus **typecheck/lint command** detection and **polyglot per-subtree**
  detection (a python root with a node subtree is no longer mislabelled). Reads
  configs statically.
- `verify-loop` Stop hook ‚Äî the feedback half of plan‚Üíwork‚Üíverify‚Üífeedback, wired
  to the repo's real verify command via the generated `.claude/harness-kit.json`.
  Non-blocking by default (reminds, does not auto-run); opt into enforcement with
  `"blocking": true`. 4-case isolation test suite.
- `new-spec` + `adr` workflow skills ‚Äî `/harness-kit:new-spec <name>` scaffolds a
  spec triplet (`specs/<date>-<name>/{spec,plan,context}.md`); `/harness-kit:adr
  <title>` records the next numbered ADR (`docs/adr/NNNN-*.md`). The generated
  spine gains a `## Workflow` section (spec discipline / ADR / scratch). Reliable
  output comes from this structured discipline, not a measurement system.
- `worktree` skill + **ask-gated** workflow ‚Äî introspect now ASKS (via
  AskUserQuestion) whether the repo wants a worktree-per-task workflow; if yes the
  spine gets a worktree rule and `.claude/harness-kit.json` records
  `worktree_workflow: true`. `/harness-kit:worktree <slug>` creates
  `../<repo>-<slug>` on its own branch. Opinionated choices are asked, not assumed.
- `handoff` + `pickup` resume loop ‚Äî `/harness-kit:handoff` writes a resume block
  (original ask, phase, next command, in-flight decisions, don't-redo) at a
  stopping point; `/harness-kit:pickup` continues from it in a fresh session.
  Shipped only after a **discriminating validation**: across 3 trials a fresh
  session that read the handoff respected a non-obvious decision (skip one of
  three near-identical code paths) and used the right naming, while a no-handoff
  control fell into the consistency trap 3/3 ‚Äî proving the block transfers context
  the repo alone does not reveal.
- `change-verifier` read-only critic agent.
- `claim-checker` read-only critic agent + spine `¬ß0.6 No overclaim` ‚Äî falsifies a
  terminal/analytical claim (limit / sufficient / fully-solves) at the conclusion
  boundary: measured or asserted, and was the falsifier run? An independent check,
  not a hook (provenance words are forgeable ‚Äî an automated gate is 0% precision).
  Effect is marginal-but-real, not a fix; the only proven 100% falsifier is a human.
- Generic critic fleet ‚Äî `instruction-critic` (right ask? before work),
  `requirement-fidelity-critic` (spec drift from the original ask),
  `spec-reviewer` (PR vs its spec; no-ops without one), `readability-critic`
  (can a human decide from this output?), `pr-shepherd` (PR to mergeable). Because
  PR workflows vary wildly, `pr-shepherd` **discovers** the repo's actual host /
  CI / bots at runtime, honors a `pr_workflow` config (`host` / `ci` / `merge_gate`)
  for intent it can't infer, degrades gracefully when CI or a host CLI is absent,
  and never fabricates a MERGEABLE verdict when no merge gate is defined.
  Seven read-only critics now form a verification spine ‚Äî an
  independent check at each boundary (instruction ‚Üí spec ‚Üí change ‚Üí claim ‚Üí output
  ‚Üí PR). The spine's static `## Critics` section routes them on demand. Stack
  architects stay generated by introspect; harness/measurement agents stay cut.
- `protected-branch-guard` PreToolUse hook (asks before commit/push on a protected
  branch). Branches are configurable per repo via `.claude/harness-kit.json`
  `protected_branches` ‚Äî precedence: env override > repo config > built-in default.
  Now reads the branch from `CLAUDE_PROJECT_DIR` (not the hook's cwd).
- `.claude/harness-kit.json` is the single per-repo config for both hooks
  (`verify_command`, `blocking`, `protected_branches`).
- `update-block.sh` ‚Äî idempotent marked-block updater so an introspect re-run
  replaces its own `CLAUDE.md` block instead of stacking copies.
- Test suites: detection (23) + verify-loop (4) + guard (6) + update-block (4); CI
  runs every `tests/*_test.sh`.
- Plugin + marketplace manifests, MIT license, community-profile files, CI.
